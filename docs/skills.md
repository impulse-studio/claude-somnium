# Skills

Somnium can create, update, and index real
[Claude Code skills](https://docs.claude.com/en/docs/claude-code/skills)
automatically — but only at the project level. This page explains why,
where they live, how the dream agent decides to make one, and how the
deduplication works.

## What "skill" means here

A skill is just a `SKILL.md` file in a directory under `.claude/skills/`,
exactly like any other Claude Code skill. There's no Somnium-specific
format or runtime — once written, you invoke it with `/<slug>` from any
Claude Code session in the same project.

Somnium doesn't run skills, doesn't ship its own skill schema, and
doesn't add a `skill_write` MCP tool. **Skill files are written by the
dream agent only**, after a session ends. Claude Code's normal skill
loading takes over from there.

## Project-only by design

The dream agent has four output categories: `global_memory`,
`project_memory`, `project_skill`, and `claude_md_patch`. There is
**no `global_skill`** category, which is a deliberate choice:

- A skill in `~/.claude/skills/` doesn't auto-load when you're working
  in a specific repo — the user has to remember it exists and invoke
  it by name. That's friction.
- A skill in `<repo>/.claude/skills/` is right next to the code it
  applies to, gets versioned with the repo, and is shareable with the
  rest of the team via git.
- Generic procedural knowledge that genuinely applies everywhere
  ("how do I review a PR for security") is a better fit for
  `global_memory` — it gets surfaced via the `memory_search` MCP tool
  and the `UserPromptSubmit` injection, no manual `/slash` invocation
  required.

If the dream agent emits a `global_skill` despite the schema (it
shouldn't, the JSON schema rejects it), the router skips it with a
clear reason in the digest:

```
| Status  | Category     | Reason                                     |
|---------|--------------|--------------------------------------------|
| skipped | global_skill | global_skill not supported — use global_memory |
```

## When does the dream agent create a skill?

The system prompt tells the agent to use `project_skill` when it sees
a procedural pattern that:

1. Is specific to **this** project (uses local file paths, project
   conventions, internal commands).
2. Is reusable — the same procedure will be relevant in future
   sessions.
3. Can be expressed as ordered steps a future Claude can follow.

Typical examples:

- *"How to add a new API endpoint in this repo"* — copy the template,
  register the route, add a fixture, run the schema check.
- *"How to release this library"* — bump the version with the right
  tool, push the tag, watch CI, mark the GitHub Release as latest.
- *"How to debug a flaky test"* — check the seed, rerun in isolation,
  inspect the captured logs in `tmp/`.

Compare with what does **not** become a skill:

- A one-off bugfix — that's not procedural, it's an event.
- A user preference like "use type hints" — that's a `project_memory`
  or a `claude_md_patch`, not a procedure.
- A cross-cutting workflow rule that applies to every project — that's
  a `global_memory` so it surfaces via search and prompt injection.

## Where they live on disk

```
<repo>/.claude/skills/
├── add-api-endpoint/
│   └── SKILL.md
├── release-library/
│   └── SKILL.md
└── debug-flaky-test/
    └── SKILL.md
```

The directory name is the slug derived from the skill title. The slug
is what you type after `/` to invoke the skill in Claude Code.

The `SKILL.md` itself uses standard Claude Code frontmatter:

```markdown
---
name: Add API endpoint
description: (auto-generated by Somnium dream mode)
source: dream
---

Steps:
1. Copy `templates/endpoint.py` into `app/api/`.
2. Register the route in `app/api/__init__.py`.
3. Add a request/response fixture in `tests/fixtures/`.
4. Run `pytest tests/api/` and `python -m schemathesis run`.
```

You can edit this file by hand at any time. The dream agent only
overwrites it if it sees the same skill come up again in a future
session — see the deduplication section below.

## Updating an existing skill

The skill directory name is the slug of the title. **Same slug = same
directory = the existing `SKILL.md` is overwritten in place.** This
is the same dedup contract as memories.

Concretely, if your `<repo>/.claude/skills/` already contains
`add-api-endpoint/SKILL.md` and the user mentions in a new session
that the procedure now has a 5th step, the dream agent will:

1. See `"Add API endpoint"` in the existing-skills list it gets
   from Somnium.
2. Reuse the same title character-for-character.
3. Emit a `project_skill` item with the full updated procedure
   (4 old steps + 1 new step).
4. The router writes the new content to the same `SKILL.md`,
   overwriting the previous version.

If you have hand-edited a skill and want to protect it from being
overwritten, the simplest move is to rename the directory — the slug
no longer matches and the dream agent will treat it as new.

## What about indexing?

Project skills are indexed by Somnium just like project memories: the
hook reindex picks them up, and they show up in `memory_search` results
when you search. The scope value is `skill_project`, so you can filter
to skills only with `memory_search(query, scope="skills")`.

This means Claude (in a future session) can find a relevant skill via
semantic search even if the slug doesn't appear in the prompt. It
won't auto-invoke it — that's still up to you typing `/<slug>` — but
it'll surface as context.

## What if there's no project?

If Somnium can't detect a project root (no `.git`, no
`.claude/somnium/` marker) and the dream agent emits a `project_skill`
anyway, the router skips it with `reason: "no project detected"`. The
fact is not lost — the dream digest still records what was attempted,
and the agent should usually have downgraded it to `global_memory` in
that case (the system prompt tells it to).

## Cheat sheet

| You want to … | Right thing |
|---|---|
| Capture a project-specific procedure | Let the dream agent emit a `project_skill` |
| Capture a generic preference that applies everywhere | `global_memory` |
| Manually create a skill mid-session | Edit `<repo>/.claude/skills/<slug>/SKILL.md` directly with Claude — Somnium has no `skill_write` tool, but the file gets reindexed via the `PostToolUse` hook |
| Update an existing skill | Same — edit the file. The dream agent will also overwrite it on the next session if it spots the same topic |
| Stop a skill from being overwritten | Rename the directory (changes the slug) |
